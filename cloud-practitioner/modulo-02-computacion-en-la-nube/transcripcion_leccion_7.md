Resolvimos el problema de escalamiento con Amazon EC2 autoscaling. Pero ahora tenemos un pequeño problema de tráfico. ¿No es así? Veamos la situación.
Los clientes tienen tres opciones de cajero, pero parece que se están inclinando hacia mí porque soy simplemente adorable. Miren, aunque me siento halagado, ¿notaron mi nuevo corte de cabello? Esto está causando una distribución desigual de clientes por fila. Miren por aquí. Anabel y Aldo no tienen clientes y solamente están tomando selfies. Ellos también deberían haberse hecho nuevos cortes de cabello.
En lugar de eso, sería de gran ayuda si añadiéramos un anfitrión a nuestra cafetería. El anfitrión separa en la entrada y dirige a los clientes a una fila específica cuando entran. También vigila los cajeros y cuenta el número de personas en cada fila. Usando esta información en tiempo real, dirigen a los clientes a la fila más corta. Esto ayuda a distribuir uniformemente a los clientes y asegura que sean atendidos de manera eficiente y rápida.
La misma idea se aplica a su entorno de AWS cuando se trata de equilibrar el tráfico entre un grupo de instancias. No queremos instancias inactivas ni tampoco sobrecargadas. Aquí es donde introducimos el concepto de un balanceador de carga. Un balanceador de carga recibe solicitudes y las dirige hacia las instancias.
Hay muchos balanceadores de carga disponibles que funcionan muy bien en AWS. Así que si tiene un tipo favorito que ya hace exactamente lo que desea, siéntase libre de seguir usándolo. Sólo recuerde que tendrá que administrarlo, parcharlo, actualizarlo, manejar la conmutación por error y realizar el mantenimiento de él.
Sin embargo, si desea que AWS se encargue de todo eso y usted solamente configure una vez, eche un vistazo a Elastic Load Balancing o ELB. ELB está diseñado para distribuir el tráfico de red para mejorar la escalabilidad de la aplicación. La palabra "elastic" se refiere a su capacidad de escalar tanto hacia arriba como hacia abajo, según el tráfico, sin añadir costos por hora. ELB puede gestionar tanto el tráfico interno como externo hacia AWS. Ofrece diferentes estrategias de enrutamiento para garantizar una gestión eficiente del tráfico y, por lo tanto, un rendimiento óptimo de la aplicación.
Profundicemos en un ejemplo para nuestra cafetería. Tenemos un sitio web para que los clientes pidan bebidas. Este sitio web tiene varias capas de funcionalidad, como la capa de pedidos, la capa de producción, capa de almacenamiento y así sucesivamente.
Veamos la capa de pedidos y cómo se comunica con la capa de producción. En este momento, cada instancia frontend conoce a cada instancia backend. Si se inicia una nueva instancia de backend, necesita informar a cada instancia de frontend que ahora puede aceptar tráfico. Quiero decir, esto ya es bastante complicado con media docena de instancias. Imagine si tuviéramos cientos o miles de instancias en cada capa. Mantenerlas sincronizadas sería una enorme tarea.
Por eso el ELB está aquí para ayudar. Podemos usarlo para gestionar la vinculación de las instancias backend y las instancias frontend. Y como es regional, es una única URL que cada instancia de frontend utiliza para dirigirse a las instancias de backend. El ELB luego dirigirá el tráfico a la instancia de backend que tenga menos solicitudes pendientes.
Si el backend necesita escalar, inicia una nueva instancia. Después de que esa nueva instancia esté lista, le dice al ELB que está lista para recibir tráfico y así se pone a trabajar. El frontend ni siquiera necesita saber lo que está sucediendo. Todo se hace automáticamente y desacopla la arquitectura para que cada capa se considere su propia entidad, que escala según lo que considere necesario.
