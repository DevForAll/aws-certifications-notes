Ay. Lo siento. Estaba ordenando una taza de café. Hablemos sobre mensajería y colas.
En la cafetería hay cajeros tomando pedidos de los clientes y baristas preparando las órdenes. Actualmente el cajero toma el pedido, lo escribe en un papel y entrega este pedido al barista. El barista entonces toma el papel y prepara el pedido. Cuando llega el siguiente pedido, el proceso se repite. Esto funciona muy bien mientras tanto el cajero como el barista estén sincronizados.
¿Qué sucede si el cajero tomó el pedido, se giró para pasarlo al barista y el barista estaba en su descanso u ocupado con otro pedido? Bueno, ese cajero está atascado hasta que el barista esté listo para tomar el pedido y en cierto punto, probablemente el pedido será descartado para que el cajero pueda atender al siguiente cliente. Puedes ver cómo este es un proceso defectuoso, porque tan pronto como el cajero o el barista estén fuera de sincronización, el proceso se degradará. Esto causará retrasos en la recepción de pedidos y fallos para completar los pedidos.
En general, un proceso mucho mejor sería introducir algún tipo de buffer o cola en el sistema. En lugar de entregar el pedido directamente al barista, el cajero publicaría el pedido en un tablero de órdenes. Esta idea de colocar mensajes en un buffer se llama mensajería y colas. Así como nuestro cajero envía pedidos al barista, las aplicaciones envían mensajes entre sí para comunicarse.
Si las aplicaciones se comunican directamente como nuestro cajero y barista anteriormente, eso se llama estar estrechamente acoplados. Un rasgo distintivo de una arquitectura estrechamente acoplada es que si un solo componente falla o cambia, causa problemas en los otros componentes o incluso para todo el sistema. Por ejemplo, la aplicación A está enviando mensajes directamente a la aplicación B. Si la aplicación B falla y no puede aceptar esos mensajes, la aplicación A también comenzará a tener errores.
Si diseñamos la aplicación para usar una arquitectura débilmente acoplada, podría verse así. Al igual que nuestro cajero y barista, introdujimos un buffer entre los dos. En este caso, introdujimos una cola de mensajes. Los mensajes son enviados a la cola por la aplicación A y son procesados por la aplicación B. Si la aplicación B falla, la aplicación A no experimenta ninguna interrupción. Los mensajes que se están enviando aún pueden ser enviados a la cola y permanecerán allí hasta que eventualmente sean procesados. Esto es débilmente acoplado. Esto es lo que nos esforzamos por lograr con arquitecturas en AWS.
Y esto me lleva a dos servicios de AWS: Amazon Simple Queue Service o Amazon SQS y Amazon Simple Notification Service o Amazon SNS. SQS hace posible que envíes, almacenes y recibas mensajes entre componentes de software en cualquier volumen. Esto se hace sin perder mensajes o requerir que otros consumidores de mensajes estén disponibles.
Piensen los mensajes como nuestros pedidos de café y el tablero de pedidos como una cola SQS. Los mensajes tienen el nombre de la persona, el pedido de café y la hora en que ordenaron. Los datos dentro de un mensaje se llaman carga útil. Las colas SQS son donde se colocan los mensajes hasta que son procesados. Estas se escalan automáticamente, son confiables y son simples de configurar y usar.
SNS es similar, ya que también envía mensajes a servicios, pero tiene una gran distinción. Los mensajes SNS enviados no se mantienen para ser recogidos hasta que el servicio de procesamiento tenga un momento para obtenerlos. En cambio, los mensajes SNS necesitan una respuesta inmediata. Si SQS es el tablero de pedidos de café, SNS es el barista gritando "¡Un chocolate caliente espumoso para llevar!".
Además, SNS puede ser utilizado para distribuir notificaciones a usuarios finales mediante móvil, push, SMS y correo electrónico. Volviendo a nuestra cafetería, podríamos enviar una notificación cuando el pedido de un cliente esté listo. Podría ser un simple mensaje de texto. De hecho, parece que mi café está listo.
